feat: Integrate RAG (Retrieval-Augmented Generation) system with Slack bot

This commit integrates a complete RAG system into the Slack bot, enabling it to
answer user questions by searching through uploaded documents using AI-powered
semantic search and LLM generation.

## 🎯 Major Features Added

### RAG-Powered Question Answering
- Bot now answers questions from uploaded documents (PDF, TXT, MD, CSV)
- Uses semantic search with ChromaDB vector database
- Generates answers using free LLM APIs (Google Gemini, Groq)
- Provides source citations for transparency

### Intelligent Message Processing
- Automatic question detection (what, how, when, where, who, why, ?)
- Distinguishes between questions and casual conversation
- Formats responses optimally for Slack (markdown, citations)

### Admin Commands
- `@bot help` - Display available commands and usage instructions
- `@bot list-docs` - List all uploaded documents in the system
- `@bot stats` - Show RAG system statistics (chunks, models, providers)

### Auto-Document Loading
- Automatically scans and loads documents from `documents/` folder on startup
- Processes PDF, TXT, MD, and CSV files
- Chunks documents for optimal semantic search
- Stores embeddings in ChromaDB vector database

### Multi-Provider LLM Support
- Primary: Google Gemini (gemini-2.5-flash) - 1,500 free requests/day
- Fallback: Groq (llama-3.1-8b-instant, llama-3.3-70b-versatile) - 14,400 free requests/day
- Automatic failover if one provider is unavailable
- Rate limiting to stay within free tiers

### Dual Interface Support
- Works in Slack channels via @mentions
- Works in direct messages (no mention needed)
- Same RAG capabilities in both contexts

#### rag_system/core/rag_engine.py (New File - 302 lines)
**Purpose:** Main orchestrator that combines all RAG components

**Class:** `RAGEngine`

**Initialization:**
- Creates DocumentProcessor instance
- Creates ChromaVectorStore instance
- Creates LLMManager instance

**Methods:**

1. `add_document(file_path)` - Add single document
   - Processes document into chunks
   - Adds chunks to vector store

2. `add_documents_from_directory(directory_path)` - Batch add documents
   - Processes all supported files in directory
   - Adds all to vector store
   - Returns batch results

3. `answer_question(question, user_id, filter_metadata)` - Main RAG pipeline (180 lines)
   - Step 1: Search vector store for relevant chunks
   - Step 2: Extract context from search results
   - Step 3: Generate answer using LLM with context
   - Step 4: Prepare sources information with relevance scores
   - Returns answer with sources and metadata

4. `get_system_status()` - Get comprehensive status
   - Vector store statistics
   - LLM usage statistics
   - Configuration validation
   - System health check

5. `list_documents()` - List all documents
6. `delete_document(filename)` - Remove specific document
7. `clear_all_documents()` - Clear entire system

#### rag_system/core/vector_store.py (New File - 334 lines)
**Purpose:** ChromaDB integration for vector storage and semantic search

**Class:** `ChromaVectorStore`

**Initialization:**
- Initializes ChromaDB client
- Creates/loads collection
- Loads SentenceTransformer embedding model (all-MiniLM-L6-v2)

**Methods:**

1. `add_document(chunks, metadata)` - Store document chunks (65-120 lines)
   - Generates unique IDs for chunks
   - Creates embeddings using SentenceTransformer
   - Stores in ChromaDB with metadata
   - Returns storage results

2. `search(query, n_results, filter_metadata)` - Semantic search (122-180 lines)
   - Encodes query to embedding vector
   - Performs cosine similarity search
   - Returns top-K most relevant chunks
   - Includes distance scores and metadata

3. `get_collection_stats()` - Get statistics
   - Total chunks count
   - Unique files count
   - File list
   - Collection health

4. `list_documents()` - List all documents
5. `delete_document(filename)` - Delete by filename
6. `clear_collection()` - Clear all data
7. `_generate_chunk_id()` - Generate unique chunk IDs

**Technical Details:**
- Uses sentence-transformers/all-MiniLM-L6-v2 (384-dim vectors)
- Cosine similarity for search
- Batch processing for efficiency
- Metadata filtering support

#### rag_system/processors/document_processor.py (New File - 288 lines)
**Purpose:** Extract and chunk text from multiple document formats

**Class:** `DocumentProcessor`

**Methods:**

1. `process_file(file_path)` - Main entry point
   - Detects file type
   - Routes to appropriate processor
   - Chunks text
   - Returns chunks with metadata

2. `_process_pdf(file_path)` - PDF processing
   - Uses PyPDF2 to extract text
   - Handles multiple pages
   - Preserves page structure

3. `_process_text(file_path)` - Plain text processing
   - Reads TXT and MD files
   - UTF-8 encoding with fallback

4. `_process_csv(file_path)` - CSV processing
   - Uses pandas to read CSV
   - Converts to readable text format
   - Includes headers and row data

5. `_chunk_text(text, metadata)` - Smart text chunking (200-250 lines)
   - Splits by sentences to preserve context
   - Maintains chunk size limits (CHUNK_SIZE=1000)
   - Overlaps chunks (CHUNK_OVERLAP=200)
   - Adds chunk metadata (index, total)
   - Returns list of chunks with metadata

6. `process_directory(directory_path)` - Batch process directory
   - Scans for supported files
   - Processes each file
   - Returns results for all files

**Supported Formats:**
- PDF (.pdf) via PyPDF2
- Text (.txt) via built-in open()
- Markdown (.md) via built-in open()
- CSV (.csv) via pandas

**Chunking Strategy:**
- Sentence-based splitting
- Preserves semantic meaning
- Configurable size and overlap
- Metadata enrichment

#### rag_system/llm/llm_manager.py (New File - 312 lines)
**Purpose:** Multi-provider LLM management with automatic fallback

**Class:** `LLMManager`

**Initialization:**
- Validates API keys for all providers
- Sets up Gemini client
- Sets up Groq client
- Initializes usage tracking

**Provider Strategy:**
1. **Gemini (Primary)** - google.generativeai
   - Model: gemini-2.5-flash
   - 1,500 requests/day free
   - High quality responses

2. **Groq Fast (Secondary)** - Groq API
   - Model: llama-3.1-8b-instant
   - Ultra-fast inference
   - 14,400 requests/day free

3. **Groq Premium (Tertiary)** - Groq API
   - Model: llama-3.3-70b-versatile
   - Higher quality, slower
   - 14,400 requests/day free

**Methods:**

1. `generate_rag_response(question, context_chunks)` - Main generation (60-110 lines)
   - Tries providers in order
   - Automatic fallback on failure
   - Rate limiting checks
   - Returns response with metadata

2. `_try_gemini(prompt)` - Gemini API call
   - Uses generativeai SDK
   - Handles errors gracefully
   - Returns formatted response

3. `_try_groq(prompt, model)` - Groq API call
   - Uses Groq SDK
   - Model selection (fast/premium)
   - Error handling

4. `_prepare_context(context_chunks)` - Format context (120-140 lines)
   - Combines chunks into coherent context
   - Limits to MAX_CONTEXT_LENGTH
   - Numbers contexts for reference

5. `_create_rag_prompt(question, context)` - Prompt engineering (141-160 lines)
   - Creates structured prompt
   - Includes context and question
   - Optimized for accuracy

6. `_check_rate_limit(provider)` - Rate limit checking
   - Tracks usage per provider
   - Prevents exceeding limits
   - Returns True if OK to proceed

7. `get_usage_stats()` - Get usage statistics
   - Requests per provider
   - Available providers
   - Rate limit status

**Features:**
- Automatic provider failover
- Rate limiting per provider
- Context length management
- Structured prompting
- Error handling and logging
